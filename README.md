# Smart Task Scheduler

## Proposal Report

### 動機與目標
<!-- 說明為什麼想做這個專題 -->
上大學後，面對資管系的重重課業壓力，同時又擁有一顆想辦活動的心，如何做好時間管理，兼顧課業與活動是很需要技巧的。
每次面對一長串待辦事項，最困難的不是「做什麼」，而是「先做哪個」。
現有工具（如 Notion、Apple 內建的行事曆與提醒事項）可以記錄任務，但無法根據 deadline 緊迫程度、預估時間與優先順序，自動告訴你今天應該怎麼安排。
而且現有的工具我怎麼用都感覺差了一點，不夠順手。於是我決定自己寫一個！

這個專題想解決這個問題：使用者輸入任務清單，系統自動輸出一份今日建議排程，讓人一眼知道該從哪裡開始。

- 實作一個可輸入任務的 Web App，支援 deadline、預估時間、優先等級
- 設計排程引擎，以 Priority Queue + Greedy Algorithm 輸出建議任務順序
- 呈現清楚、有設計感的排程畫面（參考 Todoist 風格：色彩標記、清晰的時間區塊）


### 預期功能
<!-- 列出預計實作的功能 -->
**核心功能（Proposal → Final）**
- 任務新增：名稱、deadline、預估時數、優先等級（高 / 中 / 低）
- 排程引擎：根據 deadline 緊迫度 + 優先等級計算分數，以 max-heap 排序
- 每日排程檢視：以時間區塊顯示任務建議順序
- 衝突偵測：當任務總時數超過可用時間時發出警告
- 資料儲存：以 JSON 或 SQLite 儲存任務狀態

**延伸功能（時間允許再做）**
- 完成率統計：追蹤每週任務完成比率
- 時間估算誤差分析：比較預估與實際花費時間

### 與課程的關聯
<!-- 你的專題可能涉及哪些資料結構或演算法概念？為什麼？ -->

**Priority Queue（優先權佇列）**
排程引擎的核心。每個任務依 deadline 緊迫度與使用者設定的優先等級計算一個分數，以 max-heap 維護任務集合。每次詢問「下一個該做什麼」時，直接取堆頂元素，時間複雜度 O(log n)。這是 priority queue 最經典的應用場景之一。

**Greedy Algorithm（貪婪演算法）**
給定今日可用時數，排程器逐一從 heap 取出最高優先任務，只要該任務能塞進剩餘時間就放入排程。這是 Greedy 的直接應用，與課程中討論的 interval scheduling / knapsack 問題同屬一類。

**排序（Sorting）**
當兩個任務分數相同時，以 Shortest Job First（SJF）作為次要排序依據，優先安排時間較短的任務，最大化每日完成件數。

**資料模型設計**
任務以 dataclass 表示（名稱、deadline、時長、優先等級、完成狀態），任務集合支援新增、刪除、重新排序，涉及動態集合與 heap 操作的設計取捨。
